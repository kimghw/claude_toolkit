# session-queue — 세션 간 배치·큐·락 공용 메커니즘

본 스킬은 **어떤 작업을 수행하는가**를 정의하지 않는다. 여러 세션이 같은 작업 풀(예: 파일 집합, 아이템 목록)을 공유할 때 **중복 실행을 막고, 진행 상태를 재개 가능하게 추적하며, 자원 제약 하에서 배치로 묶어 실행**하는 **공용 메커니즘**만 정의한다. 호출 스킬은 본 스킬의 규약을 자신의 작업 단위(아이템)에 적용하고, 아이템 내부에서 수행할 실제 작업(변환·분석·전송 등)만 별도로 구현한다.

## 스킬 구조

- **§1 용어**: 세션·아이템·배치·락·큐 상태 정의.
- **§2 디렉토리 레이아웃**: `<workroot>/queue/` 하위 표준 구조.
- **§3 글로벌 락 프로토콜**: 원자 생성·상태 전이·해제·스테일·크래시 복구.
- **§4 큐 상태 전이**: `pending → working → done | failed` 원자 이동.
- **§5 스킵 판정**: 기존 산출물 존재 기반 사전 제외.
- **§6 배치 플래닝**: 용량 제약 하 Next-Fit Decreasing 패킹.
- **§7 정리 순서**: 큐·자산 → 락 해제 순 엄수.
- **§8 DO / DON'T**.
- **§9 호출 스킬 통합 체크리스트**.
- **§10 참조·전제·한계**.

## 1. 용어

- **호출 스킬(Caller)**: 본 스킬의 규약을 채택해 자기 작업에 적용하는 상위 스킬. 작업 루트 경로(`workroot`), 아이템 식별자 규칙, 아이템당 용량(`cost`), 배치 용량 상한(`batch_capacity`), 최종 산출물 경로 규칙, 락 상태 집합의 후반부(§1 락 상태의 `<terminal_phase>`)를 결정한다.
- **아이템(Item)**: 락의 최소 단위이자 배치의 최소 구성요소. 고유한 문자열 ID(`<item_id>`)로 식별되며 파일명 안전 문자(`[A-Za-z0-9._-]`)만 사용한다.
- **세션 ID(session_id)**: Claude Code가 세션마다 자동 부여하는 UUID(`~/.claude/projects/<project>/` 내 `.jsonl` 파일명). 별도 발급 절차 없이 현재 세션 ID를 그대로 쓴다. 락 소유자·세션 격리 디렉토리 식별자로 사용한다.
- **글로벌 락(Global Lock)**: `<workroot>/queue/locks/<item_id>.lock` 단일 정규 파일(디렉토리 아님). 파일 본문은 **개행 없는 단일 JSON 객체**: `{"owner":"<session_id>","state":"<state>","claimed_at":"<ISO8601>","updated_at":"<ISO8601>"}`. `O_CREAT|O_EXCL`로 원자 생성하여 아이템 단위 배타 점유를 보장한다.
- **배치(Batch)**: 동시에 또는 순차로 처리할 아이템 집합. 호출 스킬이 정의한 `batch_capacity`(합산 `cost` 상한)를 넘지 않도록 패킹된다.
- **큐 상태(Queue State)**: 아이템 작업 파일이 놓인 디렉토리가 상태를 표현한다 — `pending | working | done | failed`. 디렉토리 간 이동은 반드시 **`mv`(POSIX `rename(2)`)로 원자 처리**한다.
- **락 상태(Lock State)**: 락 파일 내부 JSON의 `state` 필드. 표준 진행: `pending → working → <terminal_phase> → (unlink)` 또는 `→ failed`. `<terminal_phase>`는 호출 스킬이 정의하는 종결 직전 단계명(예: `merging`·`uploading`·`verifying` 등). 전이는 반드시 §3.2의 atomic replace 규약으로 수행한다.
- **진행 중 상태(in-progress states)**: §3.4 스테일 판정에서 "진행 중"이란 **`pending`을 제외한 모든 비종결 상태**를 의미한다. 즉 `working`과 호출 스킬이 정의한 모든 `<terminal_phase>`를 포함한다(`failed`는 종결, `pending`은 점유 직후 미착수 상태이므로 제외).

## 2. 디렉토리 레이아웃

호출 스킬은 자신의 작업 루트 `<workroot>/`(예: `./<skill_name>_work/`)를 정하고 아래 트리를 준비한다. 디렉토리는 `mkdir -p`로 멱등 생성하며 이미 존재하면 건드리지 않는다.

```text
<workroot>/
└── queue/
    ├── locks/
    │   └── <item_id>.lock           ← 아이템 단위 글로벌 락 (단일 정규 파일, 본문은 개행 없는 단일 JSON 객체)
    └── sessions/
        └── <session_id>/            ← 세션별 격리 (session_id = 현재 세션 UUID)
            ├── pending/<item_id>/   ← 적재 대기 작업 파일
            ├── working/<item_id>/   ← 진행 중
            ├── done/<item_id>/      ← 호출 스킬의 후처리 입력 대기
            ├── failed/<item_id>/    ← 재시도 임계 초과로 분리
            └── <caller_dirs>/       ← 호출 스킬이 §9에서 선언한 세션 로컬 보조 디렉토리(예: parts/, out/, assets/)
```

- `locks/`는 **글로벌**(세션 공유). 서로 다른 세션의 오케스트레이터가 같은 락 파일을 두고 경합한다.
- `sessions/<session_id>/`는 **세션 로컬**(세션 격리). 다른 세션의 디렉토리는 절대 건드리지 않는다.
- 호출 스킬이 필요로 하는 중간 산출물·자산 디렉토리(`<caller_dirs>`)는 반드시 `sessions/<session_id>/` 하위에 두어 세션 간 간섭을 차단한다. 구체 디렉토리 목록은 호출 스킬이 §9 통합 체크리스트에 선언한다.

## 3. 글로벌 락 프로토콜

### 3.1 점유 (claim)

1. `<workroot>/queue/locks/<item_id>.lock`을 **`O_CREAT|O_EXCL|O_WRONLY`**(Python `open(path, "x")`)로 생성 시도한다. POSIX `open(2) + O_EXCL`은 같은 파일시스템·로컬 환경에서 원자적이며 파일이 이미 존재하면 `EEXIST`로 실패한다(NFS 등 한계는 §10 참조).
2. **실패(`EEXIST`)**: 다른 세션이 점유 중이거나 §3.5의 미해제 락이 남아 있다. 해당 아이템을 본 실행에서 스킵하고 다음 아이템으로 진행한다. 사용자에게 보고한다.
3. **성공**: 생성된 파일 핸들에 단일 JSON 객체 `{"owner":"<session_id>","state":"pending","claimed_at":"<ISO8601>","updated_at":"<ISO8601>"}`를 **한 번의 `write()`**로 기록한 뒤 close 한다. 단일 write는 부분 기록 가능성을 줄이기 위함이며, **빈 파일 관찰 창을 완전히 제거하지는 못한다**. 점유 직후~첫 write 완료 사이에 다른 세션이 락 파일을 read 하면 빈 본문을 볼 수 있으므로, **락 본문을 읽는 측은 빈 파일/JSON 파싱 실패를 "점유 진행 중"으로 해석**해야 한다(§3.5 참조).
4. **사용자 알림 (필수)**: 락 점유와 초기 JSON 기록이 성공적으로 완료되면, 오케스트레이터는 사용자에게 락 설정 완료 사실을 즉시 보고한다. 보고 내용에는 다음을 포함한다 — 점유한 `<item_id>` 목록(또는 개수), 락 파일 경로(`<workroot>/queue/locks/<item_id>.lock`), `session_id`, 점유 시각(`claimed_at`). 여러 아이템을 일괄 점유한 경우 건별 나열 대신 요약 형태(총 N건 + 대표 경로)로 보고할 수 있다.

### 3.2 상태 전이 (atomic replace 규약, 필수)

락 상태를 갱신할 때는 **절대로 기존 락 파일을 `open("w")`로 덮어쓰지 않는다**(중간에 truncate된 0바이트 상태가 노출된다).

1. **같은 디렉토리**에 임시 파일 `<item_id>.lock.tmp.<pid>`를 `O_CREAT|O_WRONLY`로 생성한다(다른 디렉토리 사용 시 `rename(2)`이 cross-device 오류로 실패할 수 있다).
2. 새 JSON(`owner`·`claimed_at`은 불변, `state`·`updated_at`만 갱신)을 한 번의 `write()`로 기록하고 close 한다.
3. `os.rename(<tmp>, <lockfile>)`로 **원자 교체**(atomic replace)한다. POSIX `rename(2)`은 같은 파일시스템 내에서 대상이 존재하면 원자적으로 덮어쓴다(swap이 아니라 replace).

이 규약을 지키면 락 본문을 읽는 측은 항상 "이전 JSON" 또는 "이후 JSON" 둘 중 하나의 완결 상태만 관찰한다.

### 3.3 해제 (release)

- **성공 해제**: `os.unlink(<workroot>/queue/locks/<item_id>.lock)`. 쉘에서는 `rm <workroot>/queue/locks/<item_id>.lock`. **`rm -rf` 금지**(락 파일만 삭제하여 실수로 디렉토리 트리 제거를 방지).
- **실패 종료**: 락 파일을 **삭제하지 않는다**. §3.2 atomic replace로 `state=failed`만 갱신한 채 남겨 사용자에게 수동 복구를 요청한다.

### 3.4 스테일 락 (stale)

- 락 파일의 mtime이 `stale_threshold`(기본 4시간) 이상이고 `state`가 §1의 **진행 중 상태**(즉 `pending` 제외 모든 비종결 상태)에 속하면 사용자에게 보고한다.
- §3.2 atomic replace는 매번 mtime을 갱신하므로, 살아있는 오케스트레이터가 점유한 락은 자연스럽게 "살아있음" 신호를 남긴다.
- **자동 탈취 금지**: 다른 오케스트레이터가 실제로 진행 중일 가능성을 배제할 수 없으므로, 본 스킬은 스테일 락을 자동 회수하지 않는다. 스테일 보고 시 사용자(또는 호출 스킬의 운영 절차)는 다음 증거를 수집해 회수 여부를 결정한다.
  1. 락 본문의 `owner` 세션이 현재도 활성인지(`~/.claude/projects/<project>/<owner>.jsonl`의 mtime).
  2. 해당 아이템의 `<workroot>/queue/sessions/<owner>/{pending,working,done}/<item_id>/` 잔존물 상태.
  3. 호출 스킬이 정의한 최종 산출물 경로의 부분 기록 여부.
- 회수 결정 시 사용자 또는 호출 스킬이 `os.unlink`로 락 파일을 제거하고, 잔존 큐·자산은 §3.5 절차에 따라 처리한다. 본 스킬은 자동 회수 로직을 제공하지 않는다.

### 3.5 크래시 복구

오케스트레이터가 점유 후 임의 시점에 강제 종료된 경우, 다음 잔존 패턴이 발생할 수 있다. 복구 책임은 사용자 또는 호출 스킬의 운영 절차에 있으며, 본 스킬은 판정 규약만 정의한다.

| 잔존 패턴 | 관찰 | 권장 처리 |
|---|---|---|
| 빈 락 파일 | `lock` 존재, 본문 0바이트 또는 JSON 파싱 실패 | 점유 직후 크래시. mtime이 `stale_threshold` 미만이면 진행 중 점유로 간주(대기), 초과면 §3.4 스테일로 처리 |
| `state=pending` 락만 잔존 | 락 본문 정상, 세션 디렉토리에 작업 파일 없음 | 큐 적재 전 크래시. §3.4 증거 수집 후 사용자 판단으로 `unlink` |
| `state=working`·`<terminal_phase>` 락 + 큐 잔존물 | 큐 디렉토리·세션 보조 디렉토리에 부분 산출물 | §3.4 증거 수집. 회수 결정 시 §7 정리 순서로 잔존물 제거 후 락 `unlink` |
| `state=failed` 락 잔존 | 정상 종료된 실패 상태 | 사용자가 원인 파악·수동 복구 후 `unlink`(자동 처리 금지) |

복구 작업은 **항상 §7 정리 순서(잔존 큐·자산 → 락)** 를 따른다. 락을 먼저 제거하면 다른 세션이 손상된 큐를 점유할 위험이 있다.

## 4. 큐 상태 전이

### 4.1 작업 파일

아이템 하나에 대해 호출 스킬이 정의한 단위 작업 파일(예: `<unit>.task.json`)을 `pending/<item_id>/` 하위에 적재한다. 작업 파일 스키마는 호출 스킬이 정의한다(입력 경로·출력 경로·파라미터·재시도 카운트 등).

### 4.2 전이 규칙

| 전이 | 수행 방법 | 수행 시점 |
|---|---|---|
| `pending → working` | `mv pending/<item_id> working/<item_id>`(디렉토리째) 또는 단위 파일별 `mv`. POSIX `rename(2)` 원자성(같은 파일시스템 전제). | 해당 아이템의 실제 처리 시작 직전 |
| `working → done` | 단위 파일 및 산출물 조각을 `mv`로 이동 | 호출 스킬의 후처리(파싱·메타데이터 수집) 완료 직후 |
| `working → failed` | `mv working/<item_id>/<unit> failed/<item_id>/<unit>` | 재시도 카운트가 임계를 초과한 순간 |
| `working` 내부 재투입 | 카운트 증가 후 동일 디렉토리에서 재시도 | 임계 미만 일시 실패 |

- **원자 이동 유지**: 모든 전이는 `mv` 한 번으로 수행한다. 복사 후 삭제(`cp; rm`)는 금지(중간에 동일 파일 2개가 존재하는 창이 생긴다).
- **교차 아이템 금지**: 한 아이템의 단위 파일을 다른 아이템의 큐 디렉토리로 옮기지 않는다.
- **락 상태 동기화**: §6.4 실행 주기에서 큐 `mv`와 락 `state` 갱신은 같은 단계 안에서 수행하되, 사이에 크래시가 발생하면 §3.5 잔존 패턴으로 분류된다(자동 일관화 책임은 본 스킬에 없으며 복구 절차로 해소한다).

## 5. 스킵 판정

배치·락 점유를 시도하기 **전에** 기존 산출물 존재 여부를 확인해 이미 완료된 아이템을 제외한다.

1. 호출 스킬이 정의한 **최종 산출물 경로 규칙**에 따라 `<item_id>`별 기대 산출물 경로를 계산한다.
2. `test -f <path>` 및 크기 > 0을 확인한다(호출 스킬이 여러 파일을 정의하면 전부 확인).
3. 이미 존재하면 **skipped** 목록에 추가하고 본 실행에서 제외한다.
4. 스킵 결과를 사용자에게 즉시 보고한다. 보고 채널(보조 로그 파일 등)은 호출 스킬이 §9에서 선언한다.
5. 처리 대상이 0개이면 "모두 완료됨"을 보고하고 종료한다.

**주의**: 스킵 판정은 **락 점유 전**에 수행한다. 락은 "실제 처리가 필요한 아이템"에만 잡는다. 스킵된 아이템의 락 파일은 건드리지 않는다.

## 6. 배치 플래닝

### 6.1 비용과 상한

호출 스킬은 아이템별 `cost`(정수, 예: 페이지 수·토큰 수·바이트 수), 배치당 용량 상한 `batch_capacity`, 전체 실행 상한 `total_capacity`(선택)를 정의한다.

- **아이템당 비용 상한 (절대값, 우회 불가)**: `cost > batch_capacity`인 아이템은 단일 배치에 담을 수 없으므로 사용자에게 **물리 분할**을 요청하고 해당 아이템은 이번 실행에서 제외한다.
- **전체 비용 상한 (권장값, 사용자 승인 시 초과 허용)**: 합산이 `total_capacity`를 초과하면 이하 절차를 중단하고 사용자에게 승인 요청 또는 하위 우선순위 아이템 제외를 제안한다.

### 6.2 Next-Fit Decreasing 패킹

본 스킬은 단순성·예측 가능성을 위해 **Next-Fit Decreasing(NFD)** 휴리스틱을 표준으로 채택한다(최적 패킹이 아니라 휴리스틱이다).

1. 스킵 제외·상한 검증 후 남은 아이템을 `cost` **내림차순**(큰 아이템 우선)으로 정렬한다.
2. 현재 배치를 하나 연다. 정렬된 순서로 아이템을 시도 추가한다.
   - `(현재 배치 합) + item.cost ≤ batch_capacity` → 현재 배치에 추가.
   - 초과 → 현재 배치를 확정·닫고, 새 배치를 열어 해당 아이템을 넣는다(이전에 닫힌 배치는 다시 열지 않는다).
3. 결과: 배치 리스트(각 배치 `cost` 합 ≤ `batch_capacity`).

> **참고**: 호출 스킬이 더 조밀한 패킹이 필요하면 First-Fit Decreasing(FFD; 매 아이템마다 이미 열린 모든 배치를 앞에서부터 검사) 또는 Best-Fit Decreasing으로 대체할 수 있다. 본 스킬의 다른 규약(락·큐·정리)은 패킹 전략에 의존하지 않는다.

### 6.3 큰 아이템 우선 원칙

- 큰 아이템을 먼저 점유·배치해야 작은 아이템들이 남는 자리를 채울 여지가 생긴다(휴리스틱 효율 향상).
- 대형 아이템이 마지막 배치에 몰리면 실패 시 재시도 비용이 커진다.
- 락 점유 순서와 배치 생성 순서 모두 이 원칙을 따른다.

### 6.4 불가분 원칙

- **아이템은 배치 경계를 넘어 분할되지 않는다**. 하나의 아이템에 속하는 모든 단위 작업은 반드시 같은 배치에 포함된다.
- 아이템 내부 단위 수가 배치 용량을 초과하면 §6.1의 아이템당 비용 상한 규칙에 따라 물리 분할을 요청한다.

### 6.5 배치 실행 주기

각 배치에 대해:

1. 배치 소속 아이템들의 `pending/<item_id>/` → `working/<item_id>/` 일괄 `mv`.
2. 각 락 파일의 상태를 §3.2 atomic replace로 `working`으로 갱신.
3. 호출 스킬이 정의한 실제 작업을 실행(서브에이전트 병렬 기동, 직접 처리 등).
4. 단위 작업 완료 시마다 §4.2 전이 규칙에 따라 `working → done | failed` 이동.
5. 아이템의 모든 단위가 `done/` 또는 `failed/`에 도달하면 호출 스킬의 후처리 단계로 진입 — 이때 락 상태를 호출 스킬이 정의한 `<terminal_phase>`로 §3.2 규약으로 갱신.
6. 배치의 모든 아이템이 종결(성공 해제 또는 `state=failed` 보존)된 뒤 다음 배치로 진행.

### 6.6 세션 용량 기반 조기 중단 (multi-session handoff)

한 세션이 모든 아이템을 독점하지 않고, **자기 용량만큼만 점유하고 나머지는 다른 세션/계정이 가져갈 수 있도록 pending에 남기는** 프로토콜.

1. 호출 스킬은 현재 세션의 처리 상한을 `session_capacity`(예: 토큰·파트·파일 수 합산 상한)로 정의한다. 일반적으로 `total_capacity`와 동일하지만, 여러 세션이 공유 큐를 분담할 때는 더 작게 설정한다.
2. 스킵 판정(§5) 후 남은 아이템을 §6.3 큰 아이템 우선 원칙대로 정렬하고, **순서대로 한 건씩 락 점유**를 시도한다.
3. 각 점유 시도 직전 `누적_cost + item.cost ≤ session_capacity`를 검사한다.
   - 통과 → 락 `O_CREAT|O_EXCL`로 점유.
   - 초과 → **해당 아이템과 그 이후 아이템은 락 claim을 시도하지 않고 종료**한다. 미점유 아이템은 `pending` 적재도 하지 않으며(즉 락·큐 모두 건드리지 않음), 사용자에게 "N개 아이템을 다른 세션용으로 보존함"을 보고한다.
4. 점유한 아이템들만 §6.2 NFD로 배치 패킹하여 §6.5 실행 주기로 처리한다.
5. 다른 세션(다른 계정 포함)이 나중에 기동되면 §5 스킵 판정 후 남아 있는 미점유 아이템을 같은 방식으로 자기 `session_capacity`만큼 가져간다. 락은 글로벌이므로 `EEXIST`로 자동 충돌 회피된다.

**주의**: 이 프로토콜은 "락 점유 타이밍 = 배치 실행 직전이 아닌 배치 플래닝 직후"를 전제로 한다. 락을 미루면 다른 세션과 경합이 생기고, 너무 빨리 잡으면 다른 세션 몫까지 선점하게 된다. 호출 스킬은 §9 체크리스트에 `session_capacity`와 "점유 타이밍"을 명시한다.

## 7. 정리 순서

검증을 통과한 아이템마다 **이 순서대로** 수행한다.

1. **큐 디렉토리 삭제**: `<workroot>/queue/sessions/<session_id>/{pending,working,done}/<item_id>/`를 `rm -rf`로 삭제.
2. **호출 스킬 자산 삭제**: 호출 스킬이 §9에서 선언한 `<caller_dirs>` 하위의 아이템 경로(예: `assets/<item_id>/`, `out/<item_id>.*`, `parts/<item_id>__*` 등)를 삭제.
3. **락 해제**: **마지막에** `<workroot>/queue/locks/<item_id>.lock` 파일을 `os.unlink`(또는 쉘 `rm <path>`, **`rm -rf` 금지**)로 삭제한다.

§9 통합 체크리스트는 본 절을 가리킬 때 "§7 단계 1·2"를 사용한다.

**순서 엄수 이유**: 큐·자산 정리 전에 락을 해제하면 다른 세션의 오케스트레이터가 "락은 없지만 큐는 반쯤 남은" 상태를 관찰해 손상된 큐를 점유할 위험이 있다.

### 7.1 라스트 원 클린업

- 모든 아이템 정리 후 `sessions/<session_id>/` 하위 표준 디렉토리(`pending`·`working`·`done`·`failed` 및 `<caller_dirs>`)가 모두 비어 있으면 `rmdir` 시도로 제거한다. 세션 디렉토리 자체도 비면 `rmdir`로 제거.
- `locks/`는 글로벌이므로 비어도 본 스킬에서 별도 판정한다(실패 무시).
- **다른 세션의 `sessions/<other_session_id>/`은 절대 건드리지 않는다**.

### 7.2 검증 실패 시

- **절대 삭제하지 않는다**. 실패 원인을 보고하고 해당 아이템의 큐·락을 보존한다.
- 락 파일은 §3.2 atomic replace로 `state=failed`로 갱신하여 사용자에게 수동 복구 대상임을 명시한다.

## 8. DO / DON'T

### DO

- 점유 전에 §5 스킵 판정을 수행해 이미 완료된 아이템을 배제한다.
- 아이템별로 **`open(<workroot>/queue/locks/<item_id>.lock, "x")`**(`O_CREAT|O_EXCL`)로 원자 점유한다. 실패 시 다음 아이템으로 넘어간다.
- 점유 성공 직후 **한 번의 `write()`**로 단일 JSON 객체를 기록한다(빈 파일 관찰 창은 §3.1에 따라 읽는 측이 처리).
- 락 상태 전이는 §3.2 atomic replace 규약을 사용한다(직접 덮어쓰기 금지).
- 큐 디렉토리 전이는 `mv`(POSIX `rename(2)`)로만 수행한다(같은 파일시스템 내 전제).
- 아이템 내부 단위 파일과 호출 스킬 보조 디렉토리를 모두 `sessions/<session_id>/` 안에 둔다.
- 배치 플래닝은 `cost` 내림차순 NFD(또는 호출 스킬이 명시 채택한 FFD/BFD)로 수행한다.
- 아이템당 `cost ≤ batch_capacity` 절대 상한, 전체 `cost ≤ total_capacity` 권장 상한을 사전 검증한다.
- 세션 용량(`session_capacity`) 초과분은 §6.6에 따라 **락도 잡지 않고 pending 적재도 하지 않은 채** 남겨 다른 세션/계정이 가져갈 수 있도록 한다.
- 정리 순서는 §7 단계 1·2 → 3(락 해제)을 엄수한다.
- 실패 시 해당 아이템의 큐·락을 보존하고 락 `state=failed`로 갱신 후 사용자에게 보고한다.
- 락 본문을 읽는 측은 빈 파일/JSON 파싱 실패를 "점유 진행 중"으로 해석한다(§3.1).
- 크래시 잔존물은 §3.5 표 분류에 따라 처리한다.

### DON'T

- 락 파일을 **디렉토리**로 만들지 않는다(본 스킬은 단일 정규 파일만 사용).
- 락 파일을 `open("w")`로 직접 덮어쓰지 않는다(0바이트 노출).
- 락 파일을 `rm -rf`로 삭제하지 않는다(경로 실수 방지).
- 큐 전이에 `cp; rm`을 쓰지 않는다(동일 파일 2개 노출 창).
- 스테일 락을 자동 탈취하지 않는다(살아있는 작업을 덮어쓸 위험; 회수는 사용자 판단).
- 한 아이템의 단위 작업을 두 배치로 분할하지 않는다.
- 점유하지 않은 아이템의 큐·락·자산을 건드리지 않는다.
- 다른 세션의 `sessions/<other_session_id>/`을 읽거나 수정하지 않는다.
- 실패 시 큐·락을 삭제하지 않는다.
- 정리 순서를 뒤집어 락을 먼저 해제하지 않는다(손상된 큐 노출 위험).

## 9. 호출 스킬 통합 체크리스트

호출 스킬은 본 스킬의 규약을 채택할 때 아래 항목을 자신의 SKILL.md에 명시적으로 정의한다.

- [ ] **`workroot`**: 기본값(예: `./<skill_name>_work/`) 및 사용자 오버라이드 방법.
- [ ] **아이템 ID 규칙**: 입력에서 `<item_id>`를 유도하는 결정적 규칙(파일명 안전 문자만 사용).
- [ ] **작업 파일 스키마**: `pending/<item_id>/` 하위에 놓일 단위 파일 형식(JSON 필드 정의, 재시도 카운트 포함).
- [ ] **비용 정의(`cost`)**: 아이템 크기를 어떤 단위로 측정하는지(페이지·토큰·바이트 등).
- [ ] **`batch_capacity`**: 배치당 비용 상한(절대 상한).
- [ ] **`total_capacity`**: 전체 실행 비용 상한(권장 상한, 승인 허용). 미사용 가능.
- [ ] **`session_capacity` (§6.6)**: 현재 세션이 한 실행에서 점유할 최대 비용. 초과분은 락·pending 모두 건드리지 않고 다른 세션용으로 보존. 락 점유 타이밍도 함께 명시(기본: 배치 플래닝 직후).
- [ ] **스킵 판정 규칙**: 최종 산출물 경로와 존재 판정 조건(파일 수, 최소 크기 등).
- [ ] **락 상태 집합**: `<terminal_phase>` 명칭(예: `merging`·`uploading`·`verifying`)과 의미. 필요 시 중간 상태 추가.
- [ ] **세션 로컬 보조 디렉토리(`<caller_dirs>`)**: `sessions/<session_id>/` 하위에 만들 디렉토리 목록(예: `assets/`·`out/`·`parts/`)과 각 디렉토리의 아이템 단위 경로 패턴.
- [ ] **§7 단계 1·2의 정리 대상 경로 패턴**: `<caller_dirs>` 하위 아이템 경로 템플릿.
- [ ] **패킹 전략**: NFD(기본) 또는 FFD/BFD(명시 채택 시 알고리즘 본문 인용).
- [ ] **보고 채널**: 사용자 보고 외에 보조 로그 파일(예: 프로젝트 보고서)을 사용한다면 그 경로와 규약.
- [ ] **크래시 복구 운영 절차**: §3.5 잔존 패턴별 사용자 의사결정 흐름.

## 10. 참조·전제·한계

### 표준·문서

- POSIX `open(2)`: `O_CREAT|O_EXCL`은 같은 파일시스템·로컬 환경에서 원자적 생성 보장.
- POSIX `rename(2)`: 같은 파일시스템 내에서 대상이 존재하면 원자적으로 대체(atomic replace).

### 전제

- 모든 큐 디렉토리·락 디렉토리·임시 파일은 **같은 파일시스템** 위에 있어야 한다(`rename(2)`이 cross-device 오류로 실패하지 않도록).
- 단일 호스트, 로컬 파일시스템(ext4·xfs·btrfs·apfs 등)에서 본 스킬의 원자성 보장이 성립한다.
- 호출 스킬은 같은 호스트에서 실행되는 여러 Claude Code 세션 간 조율을 가정한다.

### 한계

- **NFS·일부 네트워크 파일시스템**에서는 `O_EXCL`·`rename(2)` 원자성이 구현·버전·마운트 옵션에 따라 보장되지 않을 수 있다. 다중 호스트 환경에서 본 스킬을 채택하려면 호출 스킬이 별도 분산 락(예: 데이터베이스 row lock, etcd, redis)을 함께 도입하거나 본 스킬 규약을 채택하지 말아야 한다.
- **빈 파일 관찰 창**: §3.1에 정의된 대로 점유 직후 첫 write 완료 전 read는 빈 본문을 볼 수 있다. 본 스킬은 이를 제거하지 않고 읽는 측의 해석 규약(§3.1·§8 DO)으로 보완한다.
- 본 스킬은 **프로세스 간 조율**만 다룬다. 다중 스레드/프로세스 내부의 동기화(세션 내 병렬 제어)는 호출 스킬의 책임이다.
- 본 스킬은 **자동 크래시 회수**를 수행하지 않는다. §3.5 잔존 패턴 분류와 §7 정리 순서를 제공할 뿐이며, 회수 결정은 사용자 또는 호출 스킬의 운영 절차에 위임한다.
