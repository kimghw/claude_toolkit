# 핵심 원칙 (Core Principles)

출처: Anthropic skill-creator / SKILL.md §Core Principles

## 1. Concise is Key (간결함이 최고)

컨텍스트 윈도우는 **공유 자원**이다. Skill은 시스템 프롬프트·대화 히스토리·다른 Skill 메타데이터·실제 유저 요청과 컨텍스트를 공유한다.

**기본 가정: Claude는 이미 똑똑하다.** Claude가 모르는 컨텍스트만 추가하라. 모든 정보 조각에 다음을 자문하라:

- "Claude가 정말 이 설명을 필요로 하는가?"
- "이 문단이 토큰 비용을 정당화하는가?"

장황한 설명보다 간결한 예시를 선호한다.

## 2. Set Appropriate Degrees of Freedom (적절한 자유도 설정)

작업의 **취약성·가변성**에 맞춰 구체성 수준을 조정한다.

| 자유도 | 형식 | 사용 조건 |
|--------|------|----------|
| **High** | 텍스트 기반 지침 | 다양한 접근이 유효, 컨텍스트에 따른 결정, 휴리스틱 가이드 |
| **Medium** | 파라미터 있는 의사코드·스크립트 | 선호 패턴은 있으나 변형 허용, 설정이 동작에 영향 |
| **Low** | 파라미터 적은 구체 스크립트 | 취약하고 오류 빈발, 일관성이 필수, 특정 순서 필수 |

> 비유: Claude가 길을 탐색한다고 생각하라. 절벽 옆 좁은 다리는 난간(저자유도)이 필요하고, 열린 들판은 여러 경로(고자유도)를 허용한다.

## 3. Protect Validation Integrity (검증 무결성 보호)

서브에이전트로 스킬 동작을 검증할 수 있다. 그러나 이는 **평가 표면**이다. 목표는 스킬이 **일반화되는지** 확인하는 것이지, 다른 에이전트가 누설된 컨텍스트로 답을 재구성할 수 있는지가 아니다.

### 검증 원칙

- **원본 산출물 사용**: 예시 프롬프트, 출력, diff, 로그, 트레이스
- **최소 컨텍스트만 전달**: 작업 수행에 필요한 것만
- **누설 금지**: 의도한 답·의심되는 버그·의도한 수정·사전 결론을 전달하지 말 것 (검증이 명시적으로 요구하지 않는 한)

### 포워드 테스팅 지침

- 독립적 패스를 위해 **fresh thread** 사용
- 유저처럼 스킬과 요청을 전달
- 원본 아티팩트 전달, 자신의 결론은 전달하지 말 것
- 기대 답·의도한 수정 노출 금지
- 반복 사이에 디스크에 아티팩트 남기지 말 것 (오염 방지)

> 서브에이전트가 누설된 컨텍스트를 봐야만 성공한다면, 스킬이나 테스트 셋업을 먼저 조여야 한다.
